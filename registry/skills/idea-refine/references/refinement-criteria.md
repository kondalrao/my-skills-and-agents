# Refinement Criteria

Use this rubric during convergence. Not every criterion applies every time.

## User Value

If value is unclear, nothing else matters.

Ask:

- Who has this problem right now?
- What are they doing instead?
- Is this a frequent or painful problem?
- Would they switch from the current workaround?
- Is this a pull problem or a push problem?

Signals:

- Painkiller: acute, frequent, emotional, expensive, or workflow-blocking.
- Vitamin: nice to have, low urgency, easy to ignore.

Red flags:

- "Everyone could use this."
- "It is like X but better."
- The problem is real but too rare.

## Feasibility

Ask:

- What is the hardest technical or operational problem?
- Does the core technology already work reliably?
- Which third-party APIs, data sources, or people must cooperate?
- What is the minimum team, stack, and time to test value?
- Can users see value in days or weeks?

Red flags:

- The MVP still takes months.
- Multiple fragile dependencies must all work at once.
- The idea requires solving a novel hard research problem first.

## Differentiation

Ask:

- What does this do that alternatives do not?
- Is the difference valuable to users or only interesting to builders?
- Can a competitor copy it quickly?
- Is this a 10x difference on one important dimension?

Strong differentiation can come from:

1. A new capability.
2. A 10x improvement.
3. A newly served audience.
4. A context where existing tools fail.
5. A dramatically simpler experience.

## Assumption Audit

For each direction, classify assumptions:

- Must be true: wrong means the idea fails.
- Should be true: wrong means the approach changes.
- Might be true: useful later, not worth validating yet.

Validate the riskiest "must be true" assumption before building.

## MVP Scope

- One job, done well.
- Test the riskiest assumption first.
- Prefer a time-box to a feature list.
- Make the "Not Doing" list mandatory.
- If the MVP feels complete, it is probably too large.
